---
title: "Pre-shared keys"
description: "How to configure external pre-shared keys (PSKs) in s2n-tls for TLS 1.3, including PSK creation, custom selection logic, and security considerations."
---

s2n-tls supports pre-shared keys (PSKs) as of TLS 1.3. A PSK is a secret established out-of-band between a client and server. When both sides configure a matching PSK, the TLS handshake can skip certificate exchange and public-key operations entirely.

## When to use PSKs

PSKs are useful in two situations:

- **Performance-constrained environments** — skipping public-key operations reduces CPU usage on devices with limited processing power.
- **Existing key management** — if your system already distributes shared secrets, you can reuse that mechanism for TLS authentication without deploying a PKI.

<Tip>
  If you need resumption rather than a fully out-of-band shared secret, consider [session resumption](/features/session-resumption) instead.
</Tip>

## Security considerations

<Warning>
  A PSK must not be shared between more than one server and one client. An entity that acts as both a server and a client must not use the same PSK for both roles. See [Selfie: reflections on TLS 1.3 with PSK](https://eprint.iacr.org/2019/347.pdf) for details on this attack.
</Warning>

PSK identity values are transmitted unencrypted. Do not include confidential information in a PSK identity.

Do not derive a PSK secret from a password or other low-entropy input — this is vulnerable to dictionary attacks. See [RFC 9257 — Recommendations for External PSK Usage](https://www.rfc-editor.org/rfc/rfc9257.html#name-recommendations-for-externa) for guidance on generating secure PSK secrets.

## Configuring PSKs

Both the client and server must create and append matching PSKs to their connection before the handshake.

<Steps>
  <Step title="Allocate a PSK object">
    ```c
    struct s2n_psk *psk = s2n_external_psk_new();
    ```
  </Step>
  <Step title="Set the PSK identity">
    The identity is a unique label for the PSK. It is sent unencrypted during the handshake.

    ```c
    const uint8_t identity[] = "my-psk-identity";
    s2n_psk_set_identity(psk, identity, sizeof(identity));
    ```
  </Step>
  <Step title="Set the PSK secret">
    ```c
    const uint8_t secret[32] = { /* securely generated bytes */ };
    s2n_psk_set_secret(psk, secret, sizeof(secret));
    ```
  </Step>
  <Step title="Optionally change the HMAC algorithm">
    The default is `S2N_PSK_HMAC_SHA256`. The HMAC algorithm may influence cipher suite selection on the server.

    ```c
    s2n_psk_set_hmac(psk, S2N_PSK_HMAC_SHA384);
    ```
  </Step>
  <Step title="Append the PSK to the connection">
    Call this on both the client and server connections. PSKs appended first are preferred during negotiation.

    ```c
    s2n_connection_append_psk(conn, psk);
    s2n_psk_free(&psk);
    ```
  </Step>
  <Step title="Set the PSK mode on the config">
    External PSKs and session resumption cannot be used together. Set the PSK mode before the handshake:

    ```c
    s2n_config_set_psk_mode(config, S2N_PSK_MODE_EXTERNAL);
    ```

    To override the config setting for a specific connection:

    ```c
    s2n_connection_set_psk_mode(conn, S2N_PSK_MODE_EXTERNAL);
    ```
  </Step>
</Steps>

After a successful handshake, retrieve the negotiated PSK identity with `s2n_connection_get_negotiated_psk_identity()`.

## Custom PSK selection

By default, the server chooses the first PSK identity in its list that also appears in the client's list. Implement `s2n_psk_selection_callback` and register it with `s2n_config_set_psk_selection_callback()` to apply custom selection logic.

The callback receives the list of PSK identities offered by the client. Iterate through them and select one:

```c
int my_psk_selection_callback(struct s2n_connection *conn, void *context,
                               struct s2n_offered_psk_list *psk_list)
{
    struct s2n_offered_psk *offered_psk = s2n_offered_psk_new();

    while (s2n_offered_psk_list_has_next(psk_list)) {
        uint8_t *client_psk_id = NULL;
        uint16_t client_psk_id_len = 0;

        s2n_offered_psk_list_next(psk_list, offered_psk);
        s2n_offered_psk_get_identity(offered_psk, &client_psk_id, &client_psk_id_len);

        struct s2n_psk *psk = user_lookup_identity_db(client_psk_id, client_psk_id_len);
        if (psk) {
            /* Load the PSK onto the connection, then select it */
            s2n_connection_append_psk(conn, psk);
            s2n_offered_psk_list_choose_psk(psk_list, offered_psk);
            break;
        }
    }

    s2n_offered_psk_free(&offered_psk);
    return S2N_SUCCESS;
}
```

To reset the offered PSK list to its initial state within the callback, call `s2n_offered_psk_list_reread()`. The next call to `s2n_offered_psk_list_next()` will then return the first PSK in the list again.

To skip PSK negotiation and fall back to a full certificate-based handshake, call `s2n_offered_psk_list_choose_psk()` with a `NULL` identity.
