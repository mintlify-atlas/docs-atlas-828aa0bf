---
title: "Connection serialization"
description: "How to serialize and deserialize TLS connection state in s2n-tls for connection migration, storage, and full TLS offloading architectures."
---

Connection serialization converts the state of a live TLS connection into a byte string. You can transport that string to a different host or store it to disk, then reconstruct the connection and resume sending and receiving application data.

<Warning>
  The serialized connection contains cryptographic secrets, sequence numbers, the cipher suite, and the protocol version **in plaintext**. An attacker with access to this data can decrypt all past and future communications on the connection. s2n-tls does not apply a MAC or signature to the serialized data, so a modified blob will be deserialized and used as-is.

  You **must** both encrypt and MAC the serialized connection to provide secrecy and integrity. The simplest approach is to transport the blob over another TLS connection. If you store it in a database, shared cache, or send it over a network, protect it before storage or transmission.
</Warning>

## Use cases

Connection serialization is designed for advanced, exceptional use cases. Most applications will not need it.

### Private key offloading compared to connection-serialization offloading

<Tabs>
  <Tab title="Traditional private key offloading">
    The application server holds the public x509 certificate chain and TLS configuration. It forwards only private key operations to a separate key store, which returns a signature or decryption result.

    ```
     client           application server                  key store
                   ┌─────────────────────────┐
    ┌─────┐        │     s2n-tls config      ┼───────►┌─────────────┐
    │     ┼───────►│                         │◄───────┼ private key │
    └─────┘        │ public key (x509 chain) │    ▲   └─────────────┘
                   └─────────────────────────┘    │
                                      server sends material to sign
                                      key store returns signature
    ```

    The application server still owns the TLS configuration and the certificate chain.
  </Tab>
  <Tab title="Connection-serialization TLS offloading">
    The application server acts as a pure L4 proxy during the handshake, forwarding all bytes to the key store. The key store performs the entire TLS handshake, then serializes the resulting connection and sends it back. The application server deserializes the connection and uses it to read and write application data.

    ```
                                            key store
     client    application server   ┌──────────────────────────┐
                                    │     s2n-tls config       │
    ┌─────┐        ┌─────┐ ────────►│                          │
    │     ┼───────►│     │◄─────────┼       private key        │
    └─────┘        └─────┘     ▲    │                          │
                               │    │  public key (x509 chain) │
                               │    └──────────────────────────┘
                               │
                         server sends handshake bytes
                         key store returns serialized connection
    ```

    In this architecture the application server stores no TLS certificate material and owns no TLS configuration.
  </Tab>
</Tabs>

## Usage

<Steps>
  <Step title="Set a serialization version on the config">
    Setting a version prevents forwards-incompatible changes from causing deserialization failures. See [Serialization version deployment](#serialization-version-deployment) for guidance on choosing when to move to a new version.

    ```c
    s2n_config_set_serialization_version(config, S2N_SERIALIZED_CONN_V1);
    ```
  </Step>
  <Step title="Determine the buffer size and serialize the connection">
    After the handshake completes, query the required buffer size and write the serialized connection into it.

    ```c
    uint32_t serialized_len = 0;
    s2n_connection_serialization_length(conn, &serialized_len);

    uint8_t *buffer = malloc(serialized_len);
    s2n_connection_serialize(conn, buffer, serialized_len);
    ```
  </Step>
  <Step title="Protect and transport the serialized blob">
    Encrypt and MAC the buffer before transmitting or storing it. For example, wrap it in another TLS connection before sending to the application server.
  </Step>
  <Step title="Deserialize the connection on the receiving side">
    Call `s2n_connection_deserialize()` with the buffer to reconstruct the connection.

    ```c
    s2n_connection_deserialize(new_conn, buffer, serialized_len);
    free(buffer);
    ```

    The deserialized connection can send and receive application data immediately. Most handshake information is not preserved, and connection-level getters may not function normally.
  </Step>
</Steps>

## Serialization version deployment

Connection serialization is **backwards compatible but not forwards compatible**: a newer version of s2n-tls can deserialize older formats, but an older version cannot deserialize newer formats.

The serialization version controls which format is written. The library version controls which formats can be read. To avoid deserialization failures during a rolling deployment, every host in a fleet must support reading the new format before any host begins writing it.

<Note>
  Follow a two-phase rollout when upgrading the serialization version:
  1. Deploy the new s2n-tls library version to all hosts so that all hosts can read the new format.
  2. Update the serialization version in the config so that hosts begin writing the new format.
</Note>
