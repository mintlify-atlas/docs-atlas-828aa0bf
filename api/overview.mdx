## What the API is

The s2n-tls public API is defined entirely in a single header file: [`api/s2n.h`](https://github.com/aws/s2n-tls/blob/main/api/s2n.h). Every function, type, and constant prefixed with `S2N_API` or declared in that file is part of the stable public interface.

Functions that return `int` return `0` (`S2N_SUCCESS`) on success and `-1` (`S2N_FAILURE`) on failure. Functions that return a pointer return `NULL` on failure. When a function fails, `s2n_errno` is set to a value describing the error.

```c
#define S2N_SUCCESS  0
#define S2N_FAILURE -1
```

## Stability guarantees

All declarations in `api/s2n.h` are stable within a major version of s2n-tls. The library follows semantic versioning: API and ABI compatibility is maintained between minor and patch releases. Internal functions and structures not declared in `s2n.h` are not stable and may change without notice.

See [VERSIONING.rst](https://github.com/aws/s2n-tls/blob/main/VERSIONING.rst) for details on how s2n-tls handles version changes.

## Thread safety

s2n-tls APIs are **not thread-safe** unless explicitly documented otherwise. In particular:

- `s2n_init()` must be called exactly once before any other API call, from a single thread.
- A single `s2n_connection` object can be operated on by at most two threads simultaneously — one sender and one receiver — but neither send nor receive is atomic. If multiple senders or receivers share a connection, you must provide your own locking.
- `s2n_send()` and `s2n_recv()` are the only APIs that may safely be called from two different threads on the same connection at the same time.
- `s2n_config` objects should not be modified after being associated with a connection.
